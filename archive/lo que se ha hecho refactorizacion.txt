# Plan: Refactor Arquitectónico Seguro de SimpliaLeads Dashboard

## Summary
Refactorizar primero la app productiva: `src`, Supabase Edge Functions/migrations relacionadas y configuración de calidad. La arquitectura recomendada es **Feature-Sliced Clean Architecture con Ports & Adapters**, porque el proyecto es un dashboard React con mucho dominio comercial, integraciones externas y reportes, no un backend monolítico ni un sistema que necesite microservicios.

El objetivo es mantener el comportamiento actual, ordenar el código por responsabilidad, reducir archivos gigantes, tipar el dominio y dejar una base escalable para leads, conversaciones, scoring, reportes y sincronización.

## Key Changes
- Trabajar en una branch nueva: `codex/app-architecture-refactor`, sin tocar los documentos ni scripts no productivos salvo que bloqueen build/lint.
- Separar responsabilidades en capas:
  - `app`: providers, router, layout, configuración global.
  - `features`: pantallas y flujos por negocio: dashboard, followup, scoring, reporting, conversations.
  - `domain`: tipos, reglas puras, estados de lead, scoring, métricas, filtros, reportes.
  - `application`: casos de uso y view models.
  - `infrastructure`: Chatwoot, Supabase, IndexedDB, Excel/PDF/CSV, mappers externos.
  - `shared`: UI común, formatters, fechas, utilidades puras.
- Convertir servicios actuales en adapters:
  - `ChatwootService`, `SupabaseService`, `StorageService` pasan a interfaces/repositorios.
  - `ConversationMapper` queda como mapper DTO externo -> dominio.
  - Reglas duplicadas de canal, score, labels, fechas y reportes se centralizan.
- Dividir archivos grandes:
  - `LeadActionQueue.tsx`, `LeadScoringLayer.tsx`, `reportExport.ts`, `leadImport.ts`, `DashboardDataContext.tsx`, `useDashboardData.ts`.
  - Cada pantalla queda como composición de componentes pequeños + hooks/use-cases.
- Crear tipos de dominio reemplazando `any` progresivamente:
  - `Lead`, `Conversation`, `ConversationMessage`, `Inbox`, `LeadStage`, `TagConfig`, `DashboardMetrics`, `ReportDefinition`, `ReportRun`.
- Mantener comentarios técnicos en inglés. La UI visible puede seguir en español porque es texto de producto para usuarios.

## Implementation Sequence
- Phase 1: Safety baseline
  - Crear branch.
  - Registrar estado inicial: build pasa, `lint` falla con 304 errores, `tsc` falla con errores reales de tipos.
  - Añadir scripts mínimos: `typecheck`, `check`.
  - No activar `strict` todavía; primero estabilizar.
- Phase 2: Domain foundation
  - Crear `src/domain` con tipos y reglas puras.
  - Extraer parseo de fechas, labels, canales, monto, lead stage, scoring y métricas base.
  - Añadir tests unitarios para esas reglas.
- Phase 3: Data/application layer
  - Reemplazar `DashboardDataContext` gigante por provider delgado + use-cases.
  - Mantener React Query como patrón principal de fetching/caching.
  - Definir repositorios: Chatwoot, Supabase, LocalStorage/IndexedDB.
- Phase 4: Feature slices
  - Refactorizar por prioridad: dashboard overview, follow-up queue, scoring, reporting, conversations.
  - Cada feature tendrá `components`, `hooks`, `model/viewModel` y `index`.
  - No cambiar UX visible salvo correcciones necesarias.
- Phase 5: Quality hardening
  - Reducir errores de lint por dominio productivo.
  - Corregir errores de `tsc`.
  - Añadir code-splitting por tabs pesados para bajar bundle principal.
  - Documentar arquitectura en `ARCHITECTURE.md`.

## Patterns To Apply
- Repository/Adapter para Chatwoot, Supabase e IndexedDB.
- Mapper para transformar APIs externas a modelos internos.
- Strategy para lead stage, scoring thresholds y perfiles de reportes.
- Builder/Factory para exportes Excel/PDF/CSV.
- Facade para consumo de datos del dashboard desde UI.
- ViewModel/Presenter para que las pantallas no calculen reglas de negocio.

## Test Plan
- Unit tests para reglas puras: stage, score, amount parsing, channel resolution, label state, report rows.
- Typecheck: `npx tsc -p tsconfig.app.json --noEmit`.
- Lint: `npm run lint`, inicialmente como baseline y luego como gate.
- Build: `npm run build`.
- Manual QA de tabs críticos: Estrategia, Embudo, Operación, Seguimiento, Calidad, Conversaciones, Reportes.
- Verificar exportes Excel/PDF/CSV con datos vacíos y datos reales.

## Assumptions
- Primera fase excluye documentos `.docx`, `.pdf`, assets de manuales y scripts de generación documental.
- No se cambian migraciones históricas ya aplicadas; nuevas migraciones solo si aparece una necesidad real.
- No se reescribe desde cero: se migra por partes para proteger el negocio.
- Los textos visibles para el usuario siguen en español; comentarios y nombres técnicos nuevos van en inglés.

architecture.md
SimpliaLeads Dashboard Architecture
Target Style
The app now moves toward a feature-sliced clean architecture with ports and adapters. The goal is to protect the commercial domain from UI and infrastructure details while keeping the current product behavior stable.

Layers
src/app: app providers, router, route-level loading boundaries, and global composition.
src/features: business-facing slices such as dashboard, follow-up, scoring, reporting, and conversations.
src/domain: pure types and business rules for leads, stages, scoring, channels, reports, and dashboard metrics.
src/application: use-case contracts and ports consumed by the app layer.
src/infrastructure: adapters for Chatwoot, Supabase, IndexedDB, mapping, and export mechanisms.
src/shared: cross-cutting helpers with no business ownership.
Rules
UI components should render view models and dispatch intents; they should not own integration details.
Domain modules must stay pure and testable: no React, no Supabase client, no browser APIs.
Infrastructure adapters translate external payloads into domain/application contracts.
New technical comments and code identifiers should use English. User-visible copy can remain Spanish.
Refactors should be incremental and behavior-preserving unless a product change is explicitly requested.
Current Baseline
Production build compiles.
Type checking now has a dedicated script and should be kept green before deeper strictness is enabled.
Lint is configured as a migration gate: existing any usage is warning-level while new domain code should prefer explicit types.
Next Refactor Targets
Move dashboard metric calculation from useDashboardData into application/domain modules.
Split LeadActionQueue, LeadScoringLayer, reportExport, and leadImport into feature-local components, hooks, and builders.
Replace direct service imports from UI with application ports and infrastructure adapters.
Consolidate duplicated channel, score, label, and report logic between frontend and Supabase Edge Functions.
